# S03-tui-chrome-rework — reachability artefact (real TUI, PTY capture + terminal emulation)
#
# HOW: bin/sworn was built from track/2026-07-01-render-drift-reconciliation/T2-tui HEAD and
# run on a real pseudo-terminal (Python pty) whose window size was set via TIOCSWINSZ, exactly
# as a real terminal reports its size. bubbletea's checkResize() turns that into the
# tea.WindowSizeMsg the TUI now stores (previously discarded). The alt-screen output stream was
# replayed through a pyte terminal emulator to reconstruct the final on-screen frame.
#
# WHAT IT PROVES (mechanism for AC-01..AC-05):
#  - A header bar renders above the two panes: 'sworn <version>  •  no release selected'
#    on the initial screen (AC-03).
#  - The releases pane, board pane, and help bar all size to the REAL reported terminal width
#    (AC-01) instead of the old hardcoded Width(30)/Width(80)/Width(110): compare the 80-col
#    and 200-col frames below — the panes and bars grow with the terminal.
#  - Long release names are ellipsis-truncated onto a SINGLE line (…), not wrapped illegibly
#    across 2-3 lines (AC-02 + Coach pin-1 decision). At 80 cols every release is one line.
#  - The bottom help bar is a single background-styled bar spanning the full width (AC-04).
#  - No frame line exceeds the reported width, so the emulator never has to wrap TUI output —
#    the spec-identified root cause of the VS Code integrated-terminal viewport bug (AC-05).
#
# AC-05 is inherently terminal-emulator-specific (VS Code's integrated terminal). The sandbox
# cannot drive a real VS Code window; this PTY capture proves the MECHANISM, and proof.md
# carries the explicit human VS-Code smoke step, which is the accepted AC-05 form (review pin 4).
#
