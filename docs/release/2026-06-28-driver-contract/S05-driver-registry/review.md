1. [escalate] §Key design decisions D5 / spec AC-05 — enumeration advertises proxy routing that the registry's own dispatch path will not take.
   What I observed: D5 keys `Available`/`ViaProxy` on "an active proxy login (`account.Load` + `account.IsLoggedIn` with `SWORN_DIRECT` unset — the same condition `FromEnv` routes on)". That condition lives ONLY in `model.FromEnv` (config.go:63-88). The drivers this registry resolves to dispatch via `InProcess.Dispatch` → `d.newClient` = `model.NewClient` (inprocess.go:74-76, 107-110), and `NewClient` has NO proxy consult — it constructs direct-API clients from `ProviderConfig` keys. Today the loop is proxy-aware because `newAgentFromModel` calls `model.FromEnv` (internal/run/capabilities_test.go:62 documents this). The moment S06 swaps loop dispatch onto Registry.Resolve → Driver.Dispatch, proxy routing silently disappears from the loop: a logged-in keyless user sees `Available: true (via proxy)` in `sworn capabilities`, then gets a direct-API auth failure at dispatch. AC-05's ViaProxy requirement ("the routing is visible, sworn#69") and the spec's user_outcome ("the sworn-proxy path resolving through the same visible registry") both presume the proxy path survives registry resolution — but no slice owns making it survive: S05's file plan cannot touch inprocess.go (not a touchpoint), and S06's spec risks (R-01/R-02/R-03, checked) do not mention the proxy. This is the same class of cross-slice contract gap as S02's ErrKindAuth finding.
   What to ask the implementer: nothing until the Coach rules. Options for the Coach: (a) forward-note S06 (spec amendment adding an R-04: the dispatch legs must construct proxy-aware clients — e.g. inprocess gains a FromEnv-style resolver or Default(cfg) injects a proxy-aware newClient — so dispatch matches what S05's enumeration advertises), keeping S05's design as written; (b) widen S05's scope to make the in-process dispatch path proxy-aware now (touches inprocess.go, outside the declared touchpoints); (c) accept that loop dispatch goes direct-only post-S06 and change AC-05/enumeration semantics accordingly (contradicts the spec's stated user_outcome and sworn#69's visibility lesson). Option (a) preserves this slice's scope, (b) closes the gap earliest, (c) re-scopes the requirement. The Coach picks.
