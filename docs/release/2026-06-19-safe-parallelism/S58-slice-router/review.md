## Pins

1. [mechanical] §2.1 — `OracleReader` interface signature mismatch with existing `Oracle` methods
   What I observed: Design Decision 1 says "add a small interface `OracleReader` to `internal/board/oracle.go` with the two methods the router needs: `ReadSliceStatus` and `ReadBoard`." But the existing `Oracle.ReadSliceStatus` takes 7 parameters (`ctx, reader gitContentReader, trackBranch, releaseWTRef, release, sliceID, trackMap`) and `Oracle.ReadBoard` takes 4 (`ctx, reader gitContentReader, releaseRef, release`). These are not clean interface methods — they leak git-ref resolution, track-branch names, and track maps to the caller. The router would need to know about track branches, release-wt refs, and track maps to call `ReadSliceStatus`, which contradicts the spec's "no I/O except through the injected reader" and the design's own §4 "Not reading the working tree. All state comes from committed git refs via the oracle reader."
   What to ask the implementer: Define the `OracleReader` interface with router-friendly signatures that hide ref/track-map resolution — e.g. `ReadSliceStatus(ctx, release, sliceID string) (SliceState, error)` and `ReadBoard(ctx, release string) (*BoardState, error)`, with the `Oracle` holding its `gitContentReader` internally rather than receiving it per-call. Alternatively, keep the existing signatures and have the CLI (`cmd/sworn/route.go`) construct the full parameter set, passing the `Oracle` itself (not an interface) to `Route`. Either way, the interface as described in §2.1 is not directly implementable against the current code without a refactor or a wrapper. Confirm the approach before code.

2. [mechanical] §2.2 — `LastCommitTime` requires a new `git.Repo` method not in planned_files
   What I observed: Design Decision 2 says "add a `LastCommitTime(ref, path string) (int64, error)` method" and "The production `git.Repo` already has `Show` and `CatFileExists`; `LastCommitTime` wraps `git log -1 --format=%ct`." But `internal/git/git.go` has no `Log` or `LastCommitTime` method (confirmed: only Init, Branch, Checkout, Stage, Commit, RevParse, DiffRange, DiffRangeStat, Merge, Show, CatFileExists, run). Adding a method to `git.Repo` means touching `internal/git/git.go`, which is NOT in `planned_files` (S57 owns it and it's `verified`). The design's §3 file list also omits `internal/git/git.go`.
   What to ask the implementer: Add `internal/git/git.go` to `planned_files` in `status.json` (and `internal/git/git_test.go` for the test). S57 is `verified` and owns this file, but S58 is in the same track (T17) on the same branch, so the touch is additive (a new method, no modification to existing methods). Alternatively, implement `LastCommitTime` via `git.Repo.run("log", "-1", "--format=%ct", "--", path)` inside `internal/router/` using the existing `run` method if it's exported — but `run` is unexported. The cleanest path is adding `LastCommitTime` to `git.Repo` and declaring it in `planned_files`.

3. [mechanical] §2.2 — `gitContentReader` export will break `cmd/sworn/board.go` and `internal/board/oracle_test.go`
   What I observed: Design Decision 2 says "export `gitContentReader` → `GitContentReader`." The unexported `gitContentReader` interface is used in 15+ locations across `internal/board/oracle.go` (method signatures), `internal/board/oracle_test.go` (fakeGitReader, oneShotEmptyReader), and referenced in a comment in `cmd/sworn/board.go` (`oracleReader adapts *git.Repo to board.gitContentReader`). Exporting the interface is a mechanical rename, but `cmd/sworn/board.go` has its own `oracleReader` type that satisfies the unexported interface — the comment references `board.gitContentReader` which will become `board.GitContentReader`. The `oracleReader` in `board.go` will still satisfy the exported interface (same methods), so this is compile-safe, but the comment needs updating. More importantly, if `LastCommitTime` is added to the exported `GitContentReader` interface, then `oracleReader` in `cmd/sworn/board.go` will NO LONGER satisfy it (missing `LastCommitTime` method), breaking the `sworn board` command.
   What to ask the implementer: Do NOT add `LastCommitTime` to `GitContentReader` — it's a router-specific need, not a board/oracle need. Either (a) define a separate `CommitTimeReader` interface in `internal/router/` that `*git.Repo` satisfies, or (b) add `LastCommitTime` as a method on `*git.Repo` directly and have the router depend on `*git.Repo` for commit-time queries while using `OracleReader` for state reads. Adding it to `GitContentReader` breaks `cmd/sworn/board.go`'s `oracleReader`. Confirm the interface segregation approach before code.

4. [mechanical] §2.3 — `shipped` string matching is correct but `deferred` state is missing from the decision tree
   What I observed: Design Decision 3 correctly handles `shipped` as a string since `state.State` has no `Shipped` constant. However, `state.State` DOES have a `Deferred` constant (`state.Deferred = "deferred"`), and `captain-route.sh` treats `deferred` as a terminal state in the `verified` walk (line 492: `verified|shipped|deferred) : ;;`), but the spec's decision tree (§In scope) and the design's §1/§2 do not mention `deferred` at all. The bash script has no explicit `elif [[ "$SLICE_STATE" == "deferred" ]]` branch — `deferred` falls through to the `else` → `none` with "unrecognised state." The spec's decision tree also omits it. This is faithful to the bash script, but the `verified` walk's terminal-state check must include `deferred` (the bash script does at line 492). The design does not mention this.
   What to ask the implementer: Confirm the `verified` → track-walk logic treats `deferred` as terminal (skip it, same as `verified`/`shipped`), matching `captain-route.sh:492`. The top-level `deferred` state should route to `none` (matching the bash fall-through). Add a test case for `deferred` in the track walk to the parity test.

5. [mechanical] §2.5 / spec — `verification.reason` field does not exist in `state.Verification`
   What I observed: The spec says the router must do tolerant `verification.reason`: `.verification.reason` else `(.verification.violations | join("; "))` (the 2026-06-10 S06 find). The bash script reads this via `jq -r '.verification.reason // ((.verification.violations // []) | join("; "))'`. But `state.Verification` in `internal/state/state.go` has no `Reason` field — only `Result`, `VerifierSessionID`, `VerifierVerdictAt`, `VerifierWasFreshContext`, `Violations`, and `Routing`. The `Reason` field is jq-only in the bash script; it reads from the raw JSON, not a Go struct. The Go router must either (a) add a `Reason string` field to `state.Verification` (touching `internal/state/state.go`, which is NOT in planned_files and owned by S57), or (b) read the reason from `Violations` joined by `"; "` and skip the `.reason` fallback entirely (since no status.json in this repo has ever had a `reason` field — the bash script's `//` fallback always fires).
   What to ask the implementer: Check whether any `status.json` in this repo or the reference baton repo has ever set `verification.reason` as a field. If not, the Go router can simply use `strings.Join(violations, "; ")` and document that the `.reason` fallback is a bash-only jq path that was never populated in practice. If some status.json does have it, add `Reason string json:"reason,omitempty"` to `state.Verification` and add `internal/state/state.go` to `planned_files`. The S57 trial log already flagged that `verification.routing` was spec fiction that required a struct change — confirm whether `reason` is the same pattern or not.

6. [mechanical] §3 / spec — `verified` → `merge-track` vs `merge-release` requires `git merge-base --is-ancestor`, which is not in `internal/git`
   What I observed: The spec's `verified` branch says "decide `merge-track` (peers in flight, or this track unmerged into release-wt by git-ancestry) vs `merge-release` (all slices terminal AND all tracks merged)." The bash script implements this at line 520: `git -C "$RELEASE_WT_PATH" merge-base --is-ancestor "$t_ref" "$RELEASE_WT_BRANCH"`. But `internal/git/git.go` has no `MergeBase` or `IsAncestor` method. The design's §3 file plan does not include `internal/git/git.go`, and §2 does not mention adding this method. The router needs git ancestry checking to implement the merge-track/merge-release decision.
   What to ask the implementer: Add an `IsAncestor(repo, branch, ancestor string) (bool, error)` method to `internal/git/git.go` (wrapping `git merge-base --is-ancestor`), and add `internal/git/git.go` + `internal/git/git_test.go` to `planned_files`. Alternatively, define a `GitAncestryChecker` interface in the router package and have the CLI wire it. Either way, this git operation is required and not yet declared.

7. [mechanical] §2.2 — `approved-ack.md` presence check: working-tree file existence vs git-ref
   What I observed: The bash script checks `approved-ack.md` via `[[ -f "$APPROVED_ACK" ]]` — a working-tree file-existence check, NOT a git-ref check (line 280). The design says `LastCommitTime` will be used for commit-time queries on `design.md`, `review.md`, `decline.md`, but `approved-ack.md` is checked by presence, not commit time. The design's §4 says "Not reading the working tree. All state comes from committed git refs via the oracle reader." But the bash script's `approved-ack.md` check IS a working-tree check. The `_route_ct` function also checks `[[ -f "$2" ]]` before running `git log` — so commit-time queries are also gated on working-tree file existence. This is a semantic difference: the bash script operates on the worktree filesystem, while the design claims the router will be pure and read only committed refs.
   What to ask the implementer: The bash script's `approved-ack.md` check is a working-tree file-existence check (`-f`), and the commit-time queries (`_route_ct`) also require the file to exist in the working tree. The design claims "not reading the working tree" but the parity test must match the bash script. Resolve: either (a) use `CatFileExists` on the track branch ref for `approved-ack.md` (committed-ref check, stricter than the bash script — an uncommitted `approved-ack.md` would read as absent), or (b) use working-tree `os.Stat` for `approved-ack.md` presence (matching the bash script exactly). The spec says "route on commit time, not mere presence" for the design/decline/review trio, but `approved-ack.md` IS routed on mere presence. Confirm: `approved-ack.md` → `CatFileExists` on the track branch (committed), and `design.md`/`review.md`/`decline.md` → `LastCommitTime` on the track branch. Document the resolution.
