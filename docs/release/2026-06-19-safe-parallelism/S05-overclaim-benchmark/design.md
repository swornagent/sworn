# Design TL;DR — S05-overclaim-benchmark

## §1. User-visible change

A developer runs `sworn bench overclaim` and sees a table showing overclaim rate at N=1, N=2, and N=4 concurrent tracks on a deterministic 12-slice fixture. The rate is 0% across all three, proving the concurrent scheduler does not corrupt the verify gate under parallelism. With `--publish`, the table is written to `docs/benchmark/overclaim-concurrent-1to4.md` as a committed release artefact.

## §2. Design decisions not in spec (max 5)

1. **Mock RunSliceFn always returns nil (PASS).** `RunParallel` has fail-fast: a failing slice cancels remaining slices in other tracks, preventing full overclaim measurement. By having the mock always return nil, all 12 slices are processed. The mock internally records the ground truth and the simulated verifier verdict; overclaim/underclaim is computed from the recorded data, not from the return value. Rationale: the benchmark tests scheduler correctness (no verdict corruption), not the retry loop's error handling.

2. **Fixture slices distributed evenly across N tracks.** 12 slices / N tracks = slices per track (12, 6, 3 for N=1, 2, 4). All tracks are independent (no `depends_on`), so they all run in phase 0 of the scheduler's execution plan. Rationale: maximises concurrency within each N level.

3. **Ground truth stored in status.json `owner` field.** Each fixture slice's status.json has `owner` set to `"PASS"` or `"FAIL"`, which the mock RunSliceFn reads to determine ground truth. Rationale: avoids a separate fixture metadata file; status.json is already read by the harness and is the canonical per-slice state.

4. **Overclaim rate calculation: overclaims / total slices (not / FAIL slices).** The spec says "4 known overclaims out of 12 slices; assert calculated rate is 4/12 = 33.3%". So the denominator is total slices, not just FAIL slices. Rationale: spec is explicit on this.

5. **`--publish` writes the file but does not auto-commit.** The spec says "committed by --publish or manually after first run". Auto-committing from a subcommand is unusual for a stdlib-only CLI. The implementer session commits the artefact as part of the slice diff. Rationale: AC only checks the file exists; committing is the session's responsibility.

## §3. Files I'll touch grouped by purpose

- **`internal/bench/overclaim.go`** (new) — fixture generator (creates synthetic release dir with index.md + 12 slice dirs), benchmark harness (runs RunParallel at N=1,2,4 with mock RunSliceFn), overclaim/underclaim calculator, JSON + Markdown report formatter.
- **`internal/bench/overclaim_test.go`** (new) — TestOverclaimRateCalculation (4/12 = 33.3%), TestBenchmarkDeterministic (identical results across runs), TestZeroOverclaimWithCorrectGate (0% at all N).
- **`cmd/sworn/bench.go`** (touch) — add `overclaim` subcommand dispatch: if `args[0] == "overclaim"`, call `cmdBenchOverclaim(args[1:])`.
- **`docs/benchmark/overclaim-concurrent-1to4.md`** (new) — published artefact, generated by `sworn bench overclaim --publish`.

## §4. Things I'm NOT doing

- Not calling `verify.Run` or making live API calls — the mock RunSliceFn simulates the entire implement→verify loop.
- Not creating real git worktrees — fixture uses temp directories; `worktree_path` in index.md points to the temp dir so `RunParallel` skips worktree materialisation.
- Not measuring latency or throughput (spec: out of scope).
- Not testing N>4 (spec: out of scope for R3).
- Not modifying `RunParallel` or the scheduler — the benchmark is a consumer, not a modification.

## §5. Reachability plan

Reachability artefact: `docs/benchmark/overclaim-concurrent-1to4.md`, generated by running `sworn bench overclaim --publish`. The file is the committed proof that the benchmark ran and produced a 0% overclaim rate at N=1, 2, 4. The `cmdBenchOverclaim` function is the integration point (dispatched from `cmdBench` in `cmd/sworn/bench.go`).
