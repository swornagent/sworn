## Red evidence

The first command-level test showed that `sworn baton vendor --upstream --tag
v0.13.1` advanced the version and planner prompt but left the changed
`project-context-v1.json` schema stale. The source mapping contained no schemas,
so a successful vendor run could not establish embedded protocol parity.

Adding schema parity then exposed a second defect: the generic Markdown/script
transform altered normative JSON bytes. A fixture-level test failed until
schema files were given raw-copy semantics.

A coverage test also failed while any schema embedded by Sworn was absent from
the vendor source mapping. This closes the class of defect rather than only
mapping the one schema changed by v0.13.1.

## Test results

The following passed from the upgrade branch:

```text
go test ./...
go vet ./...
make build VERSION=0.2.0
git diff --check
gofmt -l .                         # no output
go run ./cmd/sworn baton vendor --upstream --tag v0.13.1 --check
                                      No changes — the embed matches the vendored source.
go run ./cmd/sworn baton diff <upstream-baton-checkout>
                                      In sync — embedded protocol matches pinned source.
```

All 12 schemas embedded by Sworn compare byte-for-byte with the v0.13.1 source.
`sworn doctor` reports the Baton version, pin currency, prompt currency, all 12
schema manifests, and schema-skew checks as `OK`.

The complete doctor command remains non-zero because of pre-existing repository
findings outside #119: historical board render drift, future-timestamp records,
legacy AGENTS splice warnings, and an undeclared project-context record. This
upgrade neither introduced nor concealed those findings.

## Reachability artefact

The built and installed binary reports:

```text
⚔ sworn · sworn 0.2.0
baton-protocol on Baton v0.13.1
```

The public commands above prove both vendor idempotence and equivalence to the
tagged upstream checkout through `cmd/sworn`, rather than through a leaf helper.

Both upstream installers were first run in dry-run mode, then applied:

```text
./install-codex.sh --dry-run -y
./install-claude.sh --dry-run -y
./install-codex.sh -y
./install-claude.sh -y
```

Post-install checks prove:

- eight Codex Baton skills and eight Claude Baton commands are present;
- both installations contain 16 JSON-valid schemas;
- both installed `VERSION` records report `v0.13.1`;
- the Claude planner prompt is byte-identical to upstream;
- the Codex planner prompt matches upstream after the required
  `.claude/baton` to `.codex/baton` path substitution and contains no stale
  Claude paths;
- the changed `project-context-v1.json` is byte-identical to upstream in both
  installations; and
- the bounded spec-ambiguity remediation contract is present in both planner
  prompts.

The installers do not install a root `VERSION` record. `sworn doctor
--sync-baton` was therefore run after installation for both Baton homes so their
provenance is explicit and their local protocol packages agree with the newly
installed Sworn engine. Doctor's local-sync check is `OK`.

## Delivered

- Sworn is pinned to and embeds Baton v0.13.1.
- Vendor-down covers every embedded schema and preserves normative JSON bytes.
- Command-level regression coverage prevents a pin from advancing over stale
  schemas.
- Planner regression coverage pins the v0.13.1 ambiguity-convergence contract.
- The local Codex and Claude Baton installations are refreshed and validated.
- The installed `sworn` engine reports Sworn 0.2.0 with Baton v0.13.1.

## Not delivered

- The pre-existing repository-wide doctor findings listed above are unchanged;
  they are outside #119 and remain owned by their existing boards and issue
  catalogue.
- v0.2.0 feature planning is deliberately sequenced after this prerequisite and
  is not part of the Baton upgrade commit.

## Divergence from plan

The requested version bump exposed two previously silent tooling gaps. The
vendor source map did not include schemas, and the installers did not preserve a
root version record. The first was repaired in the repository with regression
coverage; the second was made explicit in both local installations through the
engine's supported sync command. Fresh verification then caught that the first
raw-copy implementation still normalised schema newlines on write and applied
the prose transform during `baton diff`; Vendor and Diff now share one content
policy, with command-level coverage for exact schema bytes.
