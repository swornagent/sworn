## Two bugs found in the upgrade path itself

Neither was in scope; both blocked the upgrade and are fixed with regression guards.

**1. `--upstream --tag <newer>` could never succeed.** `fetch.go` compared the pinned SHA
against the resolved SHA of *whatever tag was requested*. A different tag resolves to a
different SHA **by definition**, so every intentional version bump aborted with *"the tag
may have been force-moved"*. The documented upgrade path was unusable; you could only ever
re-fetch the tag you already had.

Trust is **per-tag** (trust-on-first-use): the pin describes ONE tag and can only verify a
fetch of that tag. Fixed by making `UpstreamPin` self-describing (it now carries its `Tag`)
and applying the SHA/digest check only when the requested tag is the one the pin describes.

**2. `WriteUpstreamPin` corrupted the pin on every bump.** It wrote the SHA and digest while
*deliberately* preserving the `baton-protocol:` line ("Other lines … are preserved" — the
bug stated as intent). The pin was left claiming v0.11.0 while carrying v0.12.0's SHA and
digest. `sworn doctor` would report the wrong protocol version, and a later fetch of the tag
the pin *claimed* would compare that tag's real SHA against a pin secretly holding a
different tag's — aborting with a bogus "force-moved" error. Tag, SHA and digest are one
fact; they are now written together.

## Guard fidelity (Rule 12)

**I weakened a tamper guard, so I proved it still fires.** Both halves recorded.

*Still catches tampering* — pinned SHA corrupted for v0.12.0, re-fetch v0.12.0:
```text
baton vendor: baton: upstream SHA mismatch for pinned tag v0.12.0:
  pinned deadbeef..., resolved 357edd20... — the tag may have been force-moved; aborting
exit status 1
```

*No longer blocks a legitimate bump* — new regression test
`TestFetchUpstream_VersionBumpIsNotTampering`: pin describes `v1.0.0`, fetch `v1.1.0`,
succeeds. This is the guard that was missing; had it existed, bug 1 could not have shipped.

`TestFetchUpstream_SHAMismatch` and `TestFetchUpstream_DigestMismatch` still pass — I fixed
their *pins* to name the tag they fetch (restoring each test's stated intent: "simulates
tampered tarball"), rather than relaxing what they assert.

**Scope parity.** `TestVendorFailsOnUnmappedScriptInSource` hand-listed every source file,
so any new mapping rotted it into a meaningless "source file missing" error. It now builds
its fixture **from `batonFileMappings`**, so it cannot drift out of scope with the mapping
it is supposed to cover.

**The schema gate is live, and it proved itself.** Migrating to v0.12.0 turned two existing
`internal/gate` tests red — their fixtures were old-vocabulary (`"severity": "FAIL"`, no
`blocking`) and the new schema validation rejected them, appending a contract-violation
finding. That is the gate working: an old-shape payload is now a FAIL, not a silent misread.
Fixtures migrated to the v0.12.0 contract.

## Not delivered

- **`LLMCheckReport` is not itself emitted as an `llm-check-report-v1` record to disk.**
  The schema grades the model's *response*; the engine's report struct still carries
  sworn-native fields (`check_type`, `raw_response`). *Why:* emitting the record is a
  separate surface (proof-visibility) and would change the CLI's `--json` output shape.
  *Tracking:* this bundle. *Acknowledgement:* session wrap-up.
- **The reconciled prompts are not yet exercised against a live model.** All verification
  here is against the vendored prompt bodies and mocked verifiers. *Why:* a live run needs
  an API key and a real slice; the contract (schema + prompt) is what this slice delivers.
  *Tracking:* this bundle. *Acknowledgement:* session wrap-up — worth one live
  `sworn llm-check --check security-review` before relying on the gate.

## Divergence from plan

The plan (Phase C) was three tasks: re-vendor, load prompts, de-hardcode the header. All
three landed. The re-vendor turned out to be blocked by two bugs in the upgrade path itself
(above), which were fixed rather than worked around — a version bump that only works by
hand-editing the pin is not an upgrade path. Surfaced here rather than absorbed silently.
