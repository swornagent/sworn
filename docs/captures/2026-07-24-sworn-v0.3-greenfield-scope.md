## Outcome

Sworn v0.3 is the small autonomous engine for Baton.

It must take an approved Baton plan through Planner, Implementer design,
Captain review, resumed implementation, fresh Verifier, track composition,
fresh assembly verification, and exact Merge. It must recover after process
death, run independent tracks concurrently, expose truthful operator views,
and let each role select a runner and model without making Sworn a managed
inference service.

The target is functional parity with the useful original Coach loop, not parity
with its Bash process machinery or with the later twelve-rule artefact system.

## History reset

The existing v0.2 implementation and superseded v0.3 construction remain

The replacement `release/v0.3.0` retains normal ancestry from released `main`.
Its first implementation stage may remove most current production code. This is

## Product boundary

Baton owns:

- the five principles and five responsibilities;
- plan, design, proof, and status handoffs;
- transition validity and exact Git composition rules;
- the branch-aware read-only board;
- the role-independent `baton.driver/v1` invocation contract; and
- portable and autonomous-engine conformance cases.

Sworn owns:

- scheduling and bounded concurrency;
- runtime claims, leases, attempts, commands, receipts, and recovery;
- isolated worktree and process execution;
- concrete runner and provider adapters;
- explicit per-role runner and model selection;
- operational events, logs, usage, evaluation facts, and OpenTelemetry;
- pause, resume, cancel, retry, escalation, and interactive takeover; and
- the runtime overlay and release cockpit.

Sworn does not copy Baton's lifecycle into a second mutable state machine.
Every scheduling pass re-derives the next valid Baton operation from committed
records and refs. The runtime journal records only Sworn facts needed to make
external effects idempotent and recoverable.

## Lean architecture

The production shape is deliberately small:

```text
cmd/sworn       command parsing and process lifecycle
internal/baton  immutable RC2 package, validation, operation derivation
internal/runtime scheduler, claims, commands, recovery
internal/journal commands, effects, receipts, events, outbox
internal/gitx   repository identity, worktrees, exact composition
internal/driver common invocation contract and concrete adapters
internal/observe local measurement, OTel projection, eval export
internal/cockpit read-only runtime overlay and local operator controls
```

SQLite is the initial single-node journal. Baton records and Git refs remain
portable truth; SQLite is not their replica. A later hosted coordinator may use
a different journal implementation behind the same command/effect interface.

The scheduler loop is:

```text
derive Baton operation
  -> atomically claim track/runtime effect
  -> execute outside the scheduling transaction
  -> bind observed result idempotently
  -> re-read records and refs
```

Independent dependency-ready tracks may execute concurrently. Work within one
track is serial, and only one writer may own that track. Release-worktree
composition is serial. Assembly verification is a distinct fresh invocation
over the composed candidate.

## Driver contract

There is one role-independent driver boundary. A role configuration chooses a
driver and optional model; there is no driver implementation per role.

Every invocation binds:

- operation and role;
- repository, workspace, and immutable record inputs;
- configured model, if any;
- fresh-context and workspace-mutation requirements;
- time, output, resource, and cancellation bounds; and
- an invocation ID used for receipts and telemetry.

Every result reports:

- driver identity and observed model;
- transport outcome independently from Baton outcome;
- structured result or operational `NO_VERDICT`;
- duration and exit information;
- usage only when the provider reports it;
- cancellation and bounded diagnostic information; and
- the capabilities actually used.

Required concrete families are:

1. Codex CLI;
2. Claude Code CLI;
3. OpenAI-compatible HTTP;
4. DeepSeek as a named compatibility profile;
5. Gemini;
6. Amazon Bedrock; and
7. a deterministic fake used by every shared conformance case.

Codex and Claude retain their native agentic tool loops. Codex production
execution uses its non-interactive unrestricted-workspace mode; Verifier also
uses ephemeral/no-memory execution and a clean context.

HTTP model providers share one small allowlisted workspace-tool runtime for
model-performed implementation. Provider adapters translate authentication,
messages, structured output, tool calls, usage, cancellation, and errors; they
do not each implement an agent loop.

There are no bundled model defaults, silent provider fallbacks, role-specific
drivers, hosted credentials, provider marketplace, or managed inference.

## Journal and recovery invariants

- A command ID is accepted once and returns the same receipt on replay.
- An external effect is never retried while its prior result is uncertain.
- Claims and leases are finite and owner-bound.
- Process death cannot make an agent-written file authoritative Sworn state.
- Candidate, track, release, and target refs use expected-old-value checks.
- A transport failure cannot be turned into `FAIL`, `BLOCKED`, or `PASS`.
- A Verifier assessment is accepted only for the immutable candidate it read.
- Changed candidate, target, plan, evidence, or policy facts invalidate the old
  assessment.
- Pause and cancel stop new dispatch; they do not rewrite Baton truth.
- Operator commands are closed typed commands with durable receipts.
- Recovery and normal execution use the same command/effect paths.

## Operator surface

The Baton board remains the delivery projection. Sworn joins it with a runtime
overlay showing active workers, driver/model, duration, retries, token usage,
cost when reported, logs, and recovery state.

The first surfaces are:

- `sworn run`, `board`, `serve`, `pause`, `resume`, `cancel`, `retry`,
  `takeover`, `doctor`, and `version`;
- terminal rendering;
- a responsive embedded local WebUI with a typed event stream; and
- authenticated loopback controls that submit typed commands.

The UI may never mutate Baton records directly or claim progress absent from
the real engine. Reconnection rebuilds from durable snapshot plus event offset.

## Measurement, observability, and evaluation

Local engine facts are authoritative. Telemetry is an optional lossy
projection:

- OpenTelemetry traces and stable low-cardinality metrics over OTLP;
- disabled by default and explicitly opt-in;
- bounded asynchronous export with a no-op default;
- no effect on scheduling, verdicts, retries, integration, or exit status;
- no prompt, completion, source, diff, evidence body, filesystem path,
  credential, or raw CLI argv exported by default; and
- high-cardinality run and candidate identities on spans, not metric labels.

One process activation creates a trace segment correlated with the durable run
ID. Work and effect attempts are child spans. A restart creates a new segment;
it does not pretend a span survived a crash.

The local evaluation corpus stores reproducible scenario inputs and exact
outcomes. It measures:

- delivery success and exact-integration rate;
- false green, false red, blocked, and no-verdict rates;
- verifier disagreement and repair effectiveness;
- elapsed and orchestration-only time;
- invocations, retries, recovery events, and protocol overhead;
- reported tokens and cost, with unknown distinct from zero; and
- outcome by Baton version, Sworn version, driver, model, and scenario.

Any OTLP-compatible backend may receive traces. Langfuse is a useful optional
destination for trace exploration, datasets, and externally computed scores;
it is not a Sworn runtime dependency or source of control truth.

## Orchestration-library decision

The v0.3 production default is custom Go orchestration over the small journal
interface.

LangChain and LangGraph add no required capability. They do not own Sworn
state, retries, checkpoints, tools, drivers, or telemetry.

Temporal is outside the local single-binary product boundary because it adds a
separate orchestration service and a second workflow control plane.

DBOS Go is the only library worth a bounded experiment because it can provide
durable steps, queues, concurrency control, cancellation, and recovery over
SQLite or PostgreSQL. It is not admitted merely because those features exist.
The experiment is implemented behind the journal/runtime interfaces and must
pass the same crash corpus as the native implementation.

DBOS is admitted only if it:

1. removes substantially more production code than it adds;
2. preserves Baton's derived lifecycle and Sworn's typed command/effect
   receipts without a second public workflow vocabulary;
3. supports single-node SQLite without another required service;
4. preserves explicit uncertain-effect reconciliation and exact Git
   compare-and-set behavior;
5. keeps binary, dependency, startup, and idle-resource budgets;
6. exports through Sworn's OTel boundary rather than controlling truth; and
7. does not make upgrades or workflow replay semantics harder to reason about
   than the native journal.

Failure of any criterion rejects the library. The spike does not block the
native walking skeleton and cannot become the architecture by accretion.

## Size and dependency budgets

- at most eight production packages for the parity release;
- target at most 15,000 non-generated production Go lines;
- mandatory architecture review at 18,000 lines;
- no production file over 700 lines without an explicit split decision;
- at most ten direct dependencies;
- target stripped binary no larger than 25 MiB;
- no provider SDK unless a measured protocol or signing requirement makes a
  small direct implementation less reliable; and
- no generated compatibility layer without a real consumer.

These are review gates, not incentives to compress readable code.

## Delivery stages

### S0 — reset, pin, and conformance seam

Replace the superseded kernel, pin the immutable Baton RC2 release after it is
published, expose version/digest, and run all portable driver and engine
fixtures through the real `sworn` test harness.

### S1 — single-track walking skeleton

Use the real journal, Git worktree boundary, fake external driver, deterministic
checks, fresh verification, exact Merge, and crash injection to deliver one
track end to end.

### S2 — native CLI drivers

Run the same corpus through Codex CLI and Claude Code CLI with clean Verifier
contexts, cancellation, bounded output, and explicit model selection.

### S3 — Coach topology and recovery

Add dependency-ready parallel tracks, one writer per track, track composition,
assembly verification, pause/resume/cancel/retry/takeover, and crash recovery at
every external-effect edge.

### S4 — HTTP and cloud drivers

Add OpenAI-compatible, DeepSeek, Gemini, and Bedrock adapters over the common
tool runtime. Pass deterministic fake-server and credential-gated live smokes.

### S5 — observability and evaluation

Add local measurements, direct OTel export, privacy controls, corpus runner,
score export, and the bounded DBOS comparison.

### S6 — cockpit

Add terminal and browser runtime overlays, typed controls, logs, evidence
drill-down, reconnection, and multi-run discovery.

### S7 — parity and release proof

Run unattended representative multi-track delivery with at least two driver
families and different role models. Inject timeout, process death, stale target,
failed verification, blocked specification, and composition conflict. Publish
the parity matrix and exact release evidence.

## Acceptance gate

v0.3 is not ready because code builds, a board animates, or one adapter returns
JSON. It is ready only when:

1. the real binary passes all Baton autonomous-engine cases;
2. a real multi-track repository reaches exact integration unattended;
3. every named driver passes the shared deterministic corpus and configured
   live smoke;
4. Verifier freshness and read-only containment are proven;
5. recovery tests cover every external-effect boundary;
6. terminal and WebUI views remain truthful during restart and failure;
7. opt-in telemetry failure cannot affect delivery;
8. the useful Coach parity matrix has no unratified gap; and
9. elapsed time, protocol tokens, artefacts, retries, and quality are measured
   against the preserved baseline.
