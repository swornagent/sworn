## 2. Implement against the ratified contract

### 2.1 Capability policy (`capability-policy-v1`) — the ADR-0013 build

Baton now declares the **hard eligibility contract**: a capability taxonomy + each role's required
capabilities. Implement **eligibility = role.requires ∩ provider-registry.provides** as the hard
gate; eval-based routing picks among the eligible (soft prior, never a gate). This is the durable
fix for **findings 2 and 3** — "read what the provider publishes and match it against a declared
requirement," not "hardcode per model." Build the provider registry (proactive metadata + cache
under `~/.config/sworn/`, reactive fallback) and the taxonomy mapping layer per ADR-0013. Baton
does NOT rank or pick — that stays engine-side.

Edge case now defined: `capability-policy` carries `on_capability_absent`
(`prose_fallback_gated` | `rule2_deferral`) for the override path where a gate is routed to a model
lacking the capability. Honour it; a silent no-op is forbidden.

### 2.2 Rule 12 — verifier guard-fidelity gate (new Gate 6b)

The vendored `verifier.md` now carries **Gate 6b**. Wherever sworn runs the verifier workflow /
`sworn verify`, enforce it: when a proof cites a guard as evidence for a domain-quantified claim,
check scope parity, confirm the guard fails against the **real** defect form (not a strawman
mutation), require a parser not a regex where detection needs structure, and reject
absence-of-known-bad in place of presence-of-truth. This is the teeth that make Rule 12 non-advisory.

### 2.3 Rule 10 — mock detector is a code construct, not a string match

Refine the first-pass boundary-mock detector to match **code tokens only** — AST-level, or a lexer
that skips string-literal and comment spans. This directly fixes **finding 5**: the assemble slice
that failed its own gate because a string literal `"// @no-mock\n// @mock-boundary …"` matched the
text detector. A mock is a call/binding that substitutes the boundary, not the appearance of the
word.

### 2.4 Rule 11 — resume restores the worktree before re-dispatch

On resume after an unclean exit, `reset --hard` + `clean` each track worktree to its committed slice
state, **target asserted first** (Rule 11 clause 2 — a `reset --hard` in the wrong directory is the
high-blast-radius case), before re-dispatching. Fixes **finding 1**: crash debris contaminated the
retry's diff (and tripped 2.3's detector on leftover code). Preserve committed progress + the board;
discard only uncommitted leftovers.

### 2.5 Rule 9 — autonomous design authority

Honour the `autonomous_design_authority` governance setting (project config, e.g.
`docs/baton/design-fidelity.json`): default `hard_pause` on Type-1 (halt + page, never
auto-proceed); `auto_proceed_recorded` only when the operator has explicitly set it, and then the
loop records the Type-1 decision attributed to the standing delegation. The setting is human-set —
sworn must not let an agent set it for itself. Fixes **finding 6** (autonomous loop proceeding past
an ungated design decision).

### 2.6 Rule 8 — bounded AC (advisory)

Optional `reqverify` / `specquality` addition: flag an AC that quantifies over an open domain
(no enumerable named set) as not-verifiable. Advisory until you choose to gate it.

## Addendum (2026-07-12): cutover QA runbook — post-v0.11.0

Landed on baton `main` **after** the v0.11.0 tag (baton PR #67, closes baton#66; rules-content **0.7.1**, package version unchanged). **Heads-up for the in-flight revendor:** if you are vendoring from the `v0.11.0` tag (correct per vendor-from-tag discipline), this is **not** in it. Pick it up on your next vendor bump past v0.11.0, or ping for a `v0.11.1` tag if you want it now. Nothing below blocks the v0.11.0 revendor.

### What it is
A new **additive Rule 10 artefact** (not a rule; count stays twelve): `qa-runbook.md`, a generated per-release human-facing walk of *what changed and how to check it*, that **targets** manual QA rather than replacing it. See `baton/customer-journey-validation.md` → "The cutover QA runbook" and `baton/release-mode-template/qa-runbook.md`.

### Engine work (folds into the assemble/cutover step — §2 above, W4)
Render `qa-runbook.md` at cutover, **after the assembly run passes** — it is the guided human pass between `assembly-proof.json` (machine) and the `attestations-v1` sign-off (human). It is a **rendered view, not new data**: aggregate from records you already have —
- each verified slice's reachability smoke-step (Rule 1),
- the touched journeys (`journeys-v1`),
- each slice's `delivered` list (`proof-v1`),
- the new wire surfaces (`contracts-v1`) as spot-checks
— grouped by touched journey (what changed → how to check → expected). Prose Markdown, not a JSON record (records-vs-prose — **no `qa-runbook-v1` schema**). **Advisory**: an aid, not a new fail-closed gate; the attestation stays the gating artefact and may reference the runbook revision it was walked against.

So the cutover chain the engine emits is: `assembly-proof.json` (machine pass) → `qa-runbook.md` (guided walk) → attestation (signed). This is the natural companion to the `sworn assemble` work already on the list, not separate scope.

baton#66 — two independent arrivals at the same artefact (the dogfood's manual-QA-doesn't-scale finding + an outside practitioner generating the same runbook unprompted).
